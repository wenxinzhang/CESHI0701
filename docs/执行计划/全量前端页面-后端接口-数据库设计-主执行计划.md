### 1.2 关键结论

1. **组织 / 权限 / 认证 / 模型配置四大块已接真实后端**（前端 `VITE_USE_MOCK=false`），但**全量核对暴露出 3 处真实缺陷**（见 §1.3），并非「全部完好」。
2. **五大智能体模块已有子计划**：其中安全策略、工具权限、Skills、AG-UI **已实际完成**（子计划文档头「待执行」状态过期，以内部进度表为准）；**记忆中心仅完成阶段 0-1**，阶段 2-7 待做。
3. **运行日志（run-logs）是唯一完全空白的功能**——无后端、无表、无计划。其日志类型（对话/技能/工具/错误/记忆/系统）与已有 4 张日志表高度重叠，数据源方案是本计划的核心设计决策（见 §4、§8）。

### 1.3 全量核对暴露的既有缺陷（本计划负责修复）

| 编号 | 严重度 | 缺陷 | 现象 | 归属 |
|------|--------|------|------|------|
| G1 | **CRITICAL** | 验证码接口 `/api/captcha/image`、`/api/captcha/verify` 后端零实现 | 真实模式下 `image` 走前端 mock 兜底、`verify` 无兜底必然失败；登录验证码链路不可用 | 阶段 A |
| G2 | **HIGH** | 菜单 `update-status` 字段错位 | 基类 `updateStatus` 固定写 `status` 字段，但 `SysMenu` 表只有 `isShow` 无 `status`，切换菜单显示/隐藏必抛 Prisma 未知字段错误（500）| 阶段 A |
| G3 | **核心缺口** | ✅ 已修复 | 运行日志已建后端（方案C混合），store 接真实接口，聚合真实运行数据 | 阶段 C 完成 |
| G4 | LOW | 部门/岗位列表 `status` 过滤未生效 | 前端传 `status`，但 `pageQueryOp` 未配 `fieldEq:['status']`，条件被静默忽略 | 阶段 A（附带）|

> 架构性观察（非单点缺陷，本计划不强制处理）：`@CrudController({ api:[...] })` 的 `api` 数组是**死代码**（从未被 `getCrudOptions` 读取），基类七个 CRUD 路由（list/detail/add/update/update-status/delete/batch-delete）**无条件全部暴露**。这会造成「以为某接口未开放、实际已暴露」的认知偏差，建议后续治理时清理，本计划仅记录。

### 2.1 运行日志数据源方案（D2 已确认：方案 C 混合）

完整利弊对比见 §8。**已确认方案 C 混合**：
- 新建 `sys_agent_run_log` **只存无源表的 3 类**：conversation（对话）/ system（系统事件）/ error（错误）——由 AG-UI 对话链路与各模块异常埋点写入。
- skill / tool / memory 三类**不落新表**，查询时从已有 `sys_agent_skill_run_log` / `sys_agent_tool_call_log` / `sys_agent_memory_read_log` 聚合，归一化为统一 `AgentRunLog` 结构。
- 后端 `RunLogQueryService` 负责「新表 + 3 张聚合源」合并、统一分页/过滤/排序，对前端呈现为单一列表。

**代价**：查询逻辑最复杂（新表 + 三源 UNION 归一化 + 合并分页），需保证两套口径一致；换来的是冗余最小（只冗余无源的 3 类）。阶段 C 按此实现。

## 3. 全局技术约定（对齐现有代码）

| 项 | 约定 |
|----|------|
| 后端根目录 | `server/`（NestJS 11 + Prisma 6）|
| 无全局前缀 | `main.ts` 不设 `setGlobalPrefix`，前缀由各 controller 自带（`admin/*` 或 `api/*`）|
| CRUD 基类 | `CrudControllerBase` 自动提供 list/detail/add/update/update-status/delete/batch-delete 七接口；特殊接口在子类显式覆盖 |
| 表命名 | 模型 PascalCase + `@@map("snake_case")`；列驼峰不加 `@map`；`///` 注释同步为列 comment |
| 主键 / 公共字段 | `id Int @id @default(autoincrement())`；`tenantId Int?` + `createTime`（需更新加 `updateTime @updatedAt`）|
| Controller | `@Post` + `@Perms('action')`，继承 `BaseController`，`this.ok(data)` / `this.fail(msg)` |
| Service | 继承 `BaseService`，构造注入 `PrismaService`；单文件 ≤500 行，超即拆子 service |
| DTO | class-validator + `@ApiProperty`，枚举字段 `@IsIn` |
| 安全联动 | 复用 `SecurityCheckService.check()`（已就绪），需 import `SecurityPolicyModule` |
| 前端接口层 | `frontend/src/api/*.ts`，`import request from '@/utils/http'`，禁裸 fetch |
| schema 变更 | `cd server && npm run prisma:push`（同步注释），补迁移文件 |
| 后端验证 | `cd server && npx tsc --noEmit` |
| 前端验证 | `cd frontend && npx vite build`（含 vue-tsc）|
| **schema 漂移检查铁律** | 只能用只读 `migrate diff --from-url <DB> --to-schema-datamodel`；**绝不能拿真实开发库当 shadow db**（历史教训：曾误重置开发库清空数据，`model_provider_config` 的 API Key 需手工恢复）|

#### T1 `sys_agent_run_log`（运行日志·无源表三类专用，方案 C）

> 方案 C 下本表**只存 conversation / system / error 三类**（无已有源表的类型）；skill/tool/memory 由查询层聚合已有明细表，不写本表。表结构仍按前端 `AgentRunLog` 全字段设计，以便与聚合源归一化后字段对齐。

| 列 | 类型 | 说明 |
|----|------|------|
| id | Int PK autoincrement | 主键（前端 id 为字符串，返回时 `run-<id>` 前缀区分来源）|
| sessionId | String? @db.VarChar(64) | 会话 ID（AG-UI 对话线程标识）|
| requestId | String? @db.VarChar(64) | 请求 ID（单次运行标识）|
| agentName | String? @db.VarChar(100) | 智能体名称 |
| type | String @db.VarChar(20) | 日志类型：conversation/skill/tool/error/memory/system |
| status | String @db.VarChar(20) | 执行状态：success/failed/running/cancelled/blocked |
| summary | String @db.VarChar(500) | 内容摘要 |
| startedAt | DateTime | 开始时间 |
| endedAt | DateTime? | 结束时间（running 时为空）|
| durationMs | Int @default(0) | 总耗时（毫秒）|
| userId | Int? | 操作用户 ID（前端 user 展示用名字，可 join 或存快照）|

| sourcePage | String? @db.VarChar(100) | 来源页面 |
| skillName | String? @db.VarChar(100) | 关联技能名（type=skill 时）|
| toolName | String? @db.VarChar(100) | 关联工具名（type=tool 时）|

| refTable | String? @db.VarChar(40) | 溯源表名（如 sys_agent_tool_call_log，便于回查明细）|
| refId | Int? | 溯源记录 ID |
| detail | Json? | 按类型不同的专属详情（对话/技能/工具/错误/记忆/系统结构各异）|
| tenantId | Int? | 租户 ID |
| createTime | DateTime @default(now()) | 创建时间 |

**字段对齐说明**：完整覆盖前端 `AgentRunLog` 全部字段（§ `runLogTypes.ts`）；新增 `refTable/refId` 用于从统一日志回跳到 4 张业务明细表，避免明细字段在本表冗余膨胀；`detail` 存类型专属结构（前端 `RunLogDetailDrawer` 按 type 分派渲染）。

**方案 C 归一化映射（查询层合并 4 个来源）**：

| 前端类型 | 来源 | id 前缀 | 归一化要点 |
|----------|------|---------|-----------|
| conversation/system/error | 本表 `sys_agent_run_log` | `run-` | 直接读 |
| skill | `sys_agent_skill_run_log` | `skill-` | success→status(success/failed)，durationMs 直取，summary 由 capabilityKey 拼 |
| tool | `sys_agent_tool_call_log` | `tool-` | success→status，无 riskLevel 则 join sys_agent_tool 取；summary 由 toolKey 拼 |
| memory | `sys_agent_memory_read_log` | `mem-` | hit→status，无 durationMs 记 0；summary 由 memoryKey 拼 |

## 8. 运行日志数据源方案对比（D2，请确认）

前端 `runLogTypes.ts` 定义 6 种 `LogType`，与已有 4 张日志表的映射：

| 前端 LogType | 可映射的已有表 | 字段缺口 |
|--------------|---------------|---------|
| skill 技能调用 | `sys_agent_skill_run_log` | 无 sessionId/agentName/status 枚举/summary |
| tool 工具执行 | `sys_agent_tool_call_log` | 无 sessionId/riskLevel/status |
| memory 记忆命中 | `sys_agent_memory_read_log` | 无 durationMs/status |
| error 错误 | 无专表（散落各表 success=false）| 无统一错误源 |
| conversation 对话 | **无源表** | 完全缺 |
| system 系统事件 | **无源表** | 完全缺 |
| （安全拦截 blocked）| `sys_agent_audit_log` | 对应 status=blocked |

**前端 `AgentRunLog` 需要的统一字段**：id/sessionId/requestId/agentName/type/status/summary/startedAt/endedAt/durationMs/user/sourcePage/skillName/toolName/riskLevel/processed/detail。4 张表**均无统一的 sessionId/requestId/agentName/status/summary**，字段异构。

### 方案 A — 聚合视图（不新建表）

后端写 `RunLogAggregateService`，UNION 4 张表并归一化字段输出。
- ✅ 无冗余、数据实时一致（明细即日志）、无需各模块埋点
- ❌ **conversation/system/error 三类无源表，无法产出**（占 6 类的一半）
- ❌ UNION 异构表字段映射复杂，跨表分页/排序/关键词过滤性能差、SQL 脆弱
- ❌ 状态枚举（running/cancelled）4 表都没有，无法还原

### 方案 B — 新建统一表 `sys_agent_run_log`（推荐）

各产日志模块在咽喉位置调 `RunLogService.record` 写一条统一记录。
- ✅ 6 类全覆盖（含无源表的 conversation/system/error）
- ✅ 字段统一、分页/过滤/排序/导出简单高效，直接对齐前端
- ✅ 业务明细表继续各管各的，run_log 作可观测层，`refTable/refId` 回跳明细
- ❌ 有冗余（明细 + 日志两处）、需各模块加埋点（约 5 处）
- ❌ 埋点须 fire-and-forget，不能因日志失败阻断主流程

### 方案 C — 混合

新表只存无源表的 conversation/system/error 三类，skill/tool/memory 三类聚合已有表，查询时合并。
- ✅ 冗余最小（只冗余三类）
- ❌ 查询逻辑最复杂（新表 + 聚合视图 UNION 合并 + 归一化）、两套口径易不一致、分页更难

### 推荐与最终决策

主流程原推荐方案 B（展示层最简）。**用户最终确认方案 C 混合**（2026-07-09）：新表只存无源的 conversation/system/error，skill/tool/memory 聚合已有明细表——冗余最小，代价是查询层需做跨源归一化合并。阶段 C 按方案 C 实现（见 §4.2 归一化映射 + §阶段 C 任务清单）。

## 11. 安全红线（贯穿全程）

- 验证码 code 存 Redis 且一次性消费，不回显明文；captcha 接口 `@Public` 但限频（防刷）。
