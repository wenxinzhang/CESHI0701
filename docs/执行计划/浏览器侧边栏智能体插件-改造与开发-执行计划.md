## 二、关键架构决策（需确认的分叉点）

### 决策 A：SidePanel 聊天 UI 如何"复用现有 AG-UI 组件"？

需求强调"优先复用、不要重复造组件"。但 SidePanel 是**扩展自身页面**（`chrome-extension://`），与主前端 SPA 不同源、不同构建。三条路线：

| 方案 | 做法 | 复用度 | 复杂度 | 风险 |
| --- | --- | --- | --- | --- |
| **A1 iframe 内嵌主前端**（推荐） | SidePanel 用 `<iframe src="{VITE_AGENT_API_BASE_URL}/#/assistant?embed=sidepanel">` 加载主前端全屏聊天页；页面操作/上下文经 `postMessage` ↔ 扩展桥接 | 100%（同一套聊天组件、Markdown、模型选择、会话、流式全复用） | 中 | iframe 需登录态（Cookie/JWT）；跨域 postMessage 协议要约定清 |
| **A2 组件跨包导入** | SidePanel Vite 里 alias 到 `frontend/src`，直接 import 聊天组件 | 高 | 高 | 传递依赖爆炸（element-plus/pinia/i18n/store 全要在扩展重建）、样式与别名地狱 |
| **A3 SidePanel 内自绘轻量聊天** | 沿用现有 `FloatingAgent.vue` 的自绘思路，迁到 sidepanel 页 | 低（仅复用 shared 协议/大脑） | 中 | 与"不重复造组件"要求相悖；双份 UI 维护 |

**推荐 A1**：与项目既有的"SPA 是大脑、扩展是手"架构一致（`extension-bridge.ts`/`spa-bridge.ts` 已存在），复用度最高，且天然满足"底部模型选择复用系统配置、不写死"。SidePanel 变成一个**壳**：顶部栏 + 页面上下文卡（原生绘制，读 `chrome.tabs`）+ iframe（承载聊天）。

> 若你更看重"插件离线可独立运行、不依赖主站登录"，则选 A3。请在下方确认。

### 决策 B：AG-UI 大脑（HttpAgent 流式）跑在哪？

- **B1 沿用 offscreen brain（推荐）**：现有 `offscreen/brain.ts` 已常驻、抗 SW 回收、有完整 fetch 扛 SSE。SidePanel/Popup 只发消息，结果经 SW 回推。会话不随标签切换丢失。
- B2 直接在 SidePanel 页跑 HttpAgent：SidePanel 本身是常驻页，也有 fetch；但关闭侧栏即中断流。

**推荐 B1**（若决策 A 选 A1，则大脑其实在 iframe 内的主前端里跑，offscreen 仅用于无 iframe 的降级场景——见下方"组合"）。

### 决策组合建议（默认执行路径）

> **A1 + B(iframe内)**：SidePanel = 原生壳（顶栏+页面上下文卡+快捷操作）+ iframe（主前端 `/assistant` 全屏聊天，内含 AG-UI 大脑与全部消息 UI）。
> 扩展的"手"（页面操作/上下文提取）经 `content-script` 执行，经 SW 与 iframe 内 SPA 用既有 `extension-bridge` 协议通信。
> 悬浮体链路（FloatingAgent/offscreen brain）**保留但降级**为"未内嵌 iframe 时的兜底"，不在本期主路径删除，降低回归风险。

### 决策 C：`/assistant` 菜单页与 SidePanel 内 iframe 是否同一页面？

- 复用同一个路由页 `/assistant`，通过 query（`?embed=sidepanel`）切换"全屏工作台"与"嵌入侧栏"两套外壳密度。一份页面两种形态，零重复。

## 四、分阶段执行计划

> 每阶段末尾统一验收：`type-check` 通过、`build` 通过、控制台无报错、不破坏现有页面。

### 阶段 0 · 可行性尖刺（0.5 天）
- 验证 A1 关键假设：SidePanel 页里 `<iframe>` 能否加载主前端并保持登录态（Cookie `SameSite`/JWT 传递）。
- 若 iframe 登录态不可行 → 回退决策 A3（SidePanel 内自绘，复用 shared 协议与消息渲染逻辑）。
- 产出：一个最小 SidePanel + iframe demo，能显示主前端聊天页。**此关卡决定后续走向。**

### 阶段 1 · 主前端「智能助手」菜单与页面（0.5 天，与插件解耦，可独立验收）
1. 新增 `router/modules/assistant.ts`：一级菜单，`meta.icon: 'Headset'`（Element Plus 图标，对应耳机），`title: 'menus.assistant.title'`，路径 `/assistant`。
2. `router/modules/index.ts`：把 `assistantRoutes` 加入聚合数组（放在 permission 之后，对齐设计图"组织/权限/智能助手"顺序）。
3. `locales/langs/{zh,en}.json`：补 `menus.assistant.title = 智能助手 / AI Assistant`。
4. 新增 `views/assistant/index.vue`：薄壳页，内嵌 `art-agent-chat`（全屏态 `mode=fullscreen`），标题"AG-UI 智能体"。**不重写消息结构、不改聊天核心逻辑。**
5. 验收标准 1/2/3：菜单出现耳机图标、样式与现有菜单一致、点击进入页面且聊天可用。

### 阶段 2 · 插件 manifest 与构建骨架（0.5 天）
1. `manifest.json`：
   - `permissions` 增 `sidePanel`；
   - 增 `side_panel.default_path: "src/sidepanel/sidepanel.html"`；
   - `action` 增 `default_popup: "src/popup/popup.html"`（保留 `default_title`）；
   - `web_accessible_resources` 视 iframe 需要补充；
   - name/description/version/icons 校正（对齐"AG-UI 智能体助手 v1.0.0"）。
2. `vite.config.ts`：`rollupOptions.input` 增 `popup` 与 `sidepanel` 两个 HTML 入口（与 offscreen 并列）。
3. 建空的 `popup.html`/`sidepanel.html` + `main.ts`，先能 build 出产物、能在 Chrome 加载。
4. 验收标准 4/5/7：插件可安装、点图标弹 Popup、能打开原生右侧栏（先空壳）。

### 阶段 3 · Popup 开发（0.5 天，严格对照设计图左上角弹窗）
- 结构：蓝色方形 AG 图标 + 标题"AG-UI 智能体助手" + 描述"随时随地获取智能支持，助力高效工作"。
- 主按钮 1「打开侧边栏」（蓝色实心、满宽、右箭头）→ 发 `OPEN_SIDE_PANEL` → SW 调 `chrome.sidePanel.open({tabId})` → 关闭 Popup。
- 主按钮 2「打开当前页面」（白底描边、右外链图标）→ 读当前 tab（title/url）→ 打开侧栏并携带页面上下文。
- 下方列表：设置 / 检查更新 / 帮助与反馈（先占位，点击有反馈）。
- 底部："AG-UI 智能体助手  · v1.0.0"。
- 样式：白底、12px 圆角、1px 浅灰边、轻阴影、蓝色主色 —— 用 `shared/theme.css` 变量，与主平台一致。
- 验收标准 6：Popup 风格与系统一致。

### 阶段 4 · SidePanel 壳 + 页面上下文卡 + 快捷操作（1 天，对照百度侧栏设计图）
1. 顶栏（固定不滚动）：AG 蓝色图标 + "AG-UI 智能体" + 右侧操作（固定/新建会话/会话列表/历史/页面上下文/设置/关闭）。图标沿用系统 iconfont/Element 图标。
2. 页面上下文卡：读 `chrome.tabs.query({active})` 得 title/url/domain/favicon/状态 → 卡片展示（右上角"打开页面"图标）。监听 `chrome.tabs.onActivated/onUpdated` → 切换/刷新/改址时自动刷新（验收 10/11/20）。
3. 快捷操作：按 domain 动态给建议（百度域 → "搜索人工智能最新进展/热搜榜/新闻频道/地图"；普通页 → "总结当前页/提取主要内容/查找数据/操作当前页"）。白底蓝描边圆角（验收 16）。
4. 主体：A1 路径下嵌 `<iframe src=".../#/assistant?embed=sidepanel">`；页面上下文与快捷操作点击 → `postMessage` 注入到聊天输入或作为上下文。
5. 验收标准 8/9：侧栏原生显示不遮挡网页，百度页正常。

### 阶段 5 · 消息通信 + 页面操作打通（1.5 天）
1. 补齐 `shared/message-types.ts` 的扩展消息枚举（见第五节），全部强类型，禁止裸字符串。
2. SW 路由：Popup↔SW（开侧栏）、SidePanel↔SW↔content-script（页面操作请求-响应）、tab 变更广播 `TAB_CHANGED`/`PAGE_CONTEXT_UPDATED`。
3. content-script 剥离 UI 注入，仅保留 `PING`/`BROWSER_ACTION`/`GET_PAGE_CONTEXT` 路由（复用 `browser-actions.ts`/`locator.ts`）。页面切换后重建通信（`ensureContentScript`）。
4. 页面上下文压缩：`page-context.ts` 产出结构化 `PageContext`（headings/buttons/inputs/links/visibleTextSummary），**限长**，不发整页 DOM（第六节）。
5. 打通"帮我搜索人工智能最新进展"：Agent→`search_web`/定位输入框→输入→滚动可见→高亮"百度一下"→按配置自动点或待确认→回结果（验收 12-19）。高风险操作（删除/支付/提交/发布）走确认卡（复用 `action-confirmation.ts` 思路）。
6. 验收标准 12-20 全量走通。

### 阶段 6 · 状态持久化 + 异常处理 + 文档（1 天）
1. 状态：会话 ID/标签 ID/页面上下文/模型/侧栏开关/上下文开关/工具开关/用户设置。
   - 临时态 → `chrome.storage.session`；持久设置 → `chrome.storage.local`；会话优先接后端接口（已有 `/api/ag-ui` + agentChat store）。
   - 刷新/切标签/重开侧栏：聊天记录不丢、模型设置不丢、上下文自动刷新（验收 21）。
2. 异常处理（第七节 15 种）：chrome:// 页、商店页、未授权、content script 未就绪、通信中断、标签关闭、元素不存在/不可见/被遮挡、AI 失败、流中断、开标签失败、开侧栏失败、浏览器不支持 SidePanel —— **均在聊天区出中文友好提示，绝不静默失败**。
3. `README.md`：加载/构建/测试/权限/百度测试步骤。
4. 全量验收标准 1-25 自测；`type-check`、`build`、控制台零报错。

## 五、消息通信协议（强类型，扩展 `shared/message-types.ts`）

统一信封（沿用现有 `makeMessage`，补字段对齐需求）：

```ts
interface ExtensionMessage<T = unknown> {
  id: string
  type: ExtensionMessageType
  timestamp: number
  tabId?: number
  sessionId?: string
  payload?: T
}
```

消息类型枚举（新增/对齐需求，禁止裸字符串）：
`OPEN_SIDE_PANEL` · `OPEN_URL` · `GET_ACTIVE_TAB` · `GET_PAGE_CONTEXT` · `PAGE_CONTEXT_UPDATED` · `EXECUTE_BROWSER_ACTION` · `BROWSER_ACTION_RESULT` · `CHAT_MESSAGE` · `CHAT_STREAM_CHUNK` · `CHAT_STREAM_END` · `CHAT_ERROR` · `TAB_CHANGED` · `SESSION_CHANGED`

> 现有代码已有 `AGENT_MESSAGE/BROWSER_ACTION/AGENT_STREAM_CHUNK/...` 等类型，将做**并轨映射**（保留旧类型以兼容悬浮体兜底链路，新增上述标准类型给 popup/sidepanel 用），不破坏现有 offscreen brain 通信。

页面操作统一指令（复用现有 `browser-action-schema.ts`，对齐需求的 `BrowserAction` 联合类型）：
`OPEN_URL | CLICK | INPUT | SCROLL | EXTRACT_TEXT | GET_PAGE_CONTEXT | HIGHLIGHT`，统一返回：

```ts
interface BrowserActionResult { success: boolean; actionType: string; message: string; data?: unknown; errorCode?: string }
```

安全红线（现有已满足，继续保持）：禁 `eval`、禁执行模型返回的 JS 字符串、工具参数校验、URL 协议校验、选择器超时、元素可见/可点校验、高风险操作确认。

## 六、页面上下文压缩（`shared/page-context.ts`）

## 十一、风险与回退

- **iframe 登录态（最大风险）**：阶段 0 尖刺先验；不可行则回退 A3（SidePanel 自绘，复用 shared 协议+消息渲染），工期 +1d。
- **SidePanel API 版本**：需 Chrome≥114；`manifest.minimum_chrome_version` 现为 116，满足。
- **悬浮体链路回归**：本期保留 FloatingAgent 作兜底，不删，降低回归面；主路径切原生侧栏。
- **crxjs 多入口**：popup/sidepanel HTML 入口需在 vite input 显式登记（offscreen 已有先例）。

## 十二、本期暂不实现 / 后续建议

- 语音输入仅留入口（不接 ASR）。
- Popup 的"检查更新/帮助与反馈"先占位跳转。
- 正式版域名白名单/权限管理（当前开发期用 `http/https://*`）留后续。
- 后续可将悬浮体兜底链路彻底下线，统一到原生侧栏。
